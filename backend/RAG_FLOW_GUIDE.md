# 🤖 RAG 系统工作流程指南

> **RAG** = Retrieval Augmented Generation（检索增强生成）

## 🎯 一句话理解 RAG

**用户提问 → 系统自动在知识库找相关资料 → AI 基于资料生成回答 → 告诉用户信息来自哪里**

---

## 📖 通俗类比

想象你在图书馆查资料：

1. **用户提问**："入职信息"
   - 就像你向图书馆员说："我要找关于入职的资料"

2. **系统搜索**（向量检索）
   - 图书馆员不是傻傻地翻书，而是用一种"智能方式"快速定位相关书籍
   - 系统用数学模型（向量）来表示"入职信息"的含义
   - 然后用相似度计算找出最相关的文档

3. **整理资料**
   - 图书馆员找到 5 本相关书籍，按相关程度排序
   - 系统找到 5 个相关文档分块，按相似度排序（45%, 19%, 16%...）

4. **AI 生成回答**
   - 图书馆员把这 5 本书交给一位"智能助手"
   - 助手快速阅读，用自己的话总结答案
   - 系统把文档内容交给 LLM，生成自然流畅的回答

5. **显示来源**
   - 图书馆员告诉你："这是从第 2 和第 4 本书找到的"
   - 系统显示："📚 信息来源：company_policy.md 45% 相关度"

### 第 2️⃣ 步：意图识别

**问题：** 用户的话可能是在问知识、查数据、或者执行操作，系统怎么区分？

**解决方案：** 用大模型（LLM）分析用户意图

```
用户输入："入职流程是什么？"
  ↓
【调用 LLM 进行意图识别】
  ↓
分析关键词：
- "入职流程" → 知识类问题
- "是什么" → 询问定义/流程
- 不涉及具体数据查询或操作指令
  ↓
判断结果：knowledge_qa（知识问答）✅
```

**与其他意图的区别：**

| 用户输入 | 识别意图 | 处理方式 |
|---------|---------|---------|
| "公司请假流程是什么？" | knowledge_qa | 查知识库文档 |
| "有多少用户？" | data_query | 生成 SQL 查数据库 |
| "批准所有待审订单" | smart_operation | 执行审批操作 |
| "订单量是否正常？" | business_diagnosis | 诊断分析 |

**知识类关键词库：**
```python
knowledge_keywords = [
    "是什么", "什么是", "怎么样", "如何",
    "流程", "政策", "制度", "规定",
    "指南", "说明", "介绍", "信息",
    "为什么", "原因", "原理",
]
```

### 第 3️⃣ 步：后端生成"语义向量"

**问题：** "入职信息" 是普通文本，系统怎么理解它的意思？

**解决方案：** 用 AI 把文本转换成数学向量（一组数字）

```
"入职信息"
  ↓
【调用 Qwen 通义千问 Embedding API】
  ↓
[0.123, -0.456, 0.789, ..., 0.012]  ← 1536 个数字表示"入职信息"的含义
  ↓
【截断到 768 维】（因为数据库列是 768 维）
  ↓
[0.123, -0.456, 0.789, ..., 0.012]  ← 768 个数字
```

**类比：** 把"入职信息"这个词转换成一个"指纹"，方便快速比对相似度

---

### 第 4️⃣ 步：在数据库中搜索相似文档

**问题：** 有 N 个文档分块，怎么快速找到最相关的 5 个？

**解决方案：** 用 pgvector（PostgreSQL 的向量插件）计算相似度

```
系统把用户的向量[0.123, -0.456, ...]与数据库中的每个文档向量比对：

文档 1 的向量：[0.120, -0.450, ...] → 相似度 45% ✅ 最相似
文档 2 的向量：[0.100, -0.400, ...] → 相似度 19%
文档 3 的向量：[0.050, -0.200, ...] → 相似度 16%
文档 4 的向量：[-0.5, 0.8, ...]    → 相似度 2%  ❌ 不相关
...
```

**SQL 查询：**
```sql
SELECT dc.id, dc.chunk_text, d.filename,
       1 - (dc.embedding <=> [0.123, -0.456, ...]) as relevance
FROM document_chunks_with_vectors dc
JOIN documents d ON dc.document_id = d.id
ORDER BY relevance DESC
LIMIT 5
```

**结果（返回 5 个最相似的文档）：**
```json
[
  {
    "filename": "company_policy.md",
    "chunk_index": 0,
    "relevance": 0.454,
    "text": "### 新员工入职流程\n1. **收到 Offer 后**..."
  },
  {
    "filename": "company_policy.md",
    "chunk_index": 1,
    "relevance": 0.187,
    "text": "### 五险一金\n- 缴纳比例..."
  },
  ...
]
```

**类比：** 就像找到了 5 本最相关的书，按相关程度排序

## 📤 文档上传与向量化原理（知识库构建流程）

> 用户提问时的检索流程只是 RAG 的"使用"阶段，在此之前需要先"构建知识库"

### 🏗️ 为什么需要向量化？

**问题：** 电脑不懂"入职流程"是什么意思，怎么能找到相关文档？

**解决方案：** 把文档内容转换成数学向量，让电脑用"数学方式"理解语义

```
传统搜索：关键词匹配（只能找相同的词）
向量搜索：语义匹配（能理解同义词、近义词）

例：用户问"怎么请假"
- 关键词搜索：找不到包含"请假"的文档
- 向量搜索：能找到包含"休假""调休""事假"等相关内容的文档 ✅
```

### 🔄 文档上传处理流程（5个步骤）

```
┌─────────────────────────────────────────────────────────────────┐
│                    文档上传 → 向量化 → 入库                       │
└─────────────────────────────────────────────────────────────────┘

【第1步】用户上传文档
    ↓
    文件：company_policy.pdf / employee_handbook.md
    格式：PDF / Word / Markdown / TXT
    大小：建议 < 50MB（大文件需特殊处理）

【第2步】文本提取
    ↓
    ┌─────────────────┐
    │ 文档解析引擎      │
    │ - PDF → 提取文字  │
    │ - Word → 读取内容 │
    │ - MD → 直接读取   │
    └─────────────────┘
    ↓
    纯文本内容（可能很长：10万字+）

【第3步】文档分块（Chunking）
    ↓
    为什么分块？
    - Embedding API 有长度限制（通常 512-2048 tokens）
    - 细粒度检索更精准（能找到段落级别的相关内容）
    - 减少噪声（避免无关内容影响匹配）
    ↓
    分块策略：
    ┌─────────────────────────────────────┐
    │ 原始文档：10万字员工手册              │
    │                                     │
    │  第1章 公司介绍（5000字）            │
    │      ↓ 分成 5 块，每块 ~1000字       │
    │      ├─ chunk_0: 公司历史           │
    │      ├─ chunk_1: 企业文化           │
    │      ├─ chunk_2: 组织架构           │
    │      ...                           │
    │                                     │
    │  第2章 入职流程（3000字）            │
    │      ↓ 分成 3 块                    │
    │      ├─ chunk_5: Offer接收          │
    │      ├─ chunk_6: 入职材料           │
    │      └─ chunk_7: 第一天安排         │
    │                                     │
    │  第3章 薪酬福利（8000字）            │
    │      ↓ 分成 8 块...                 │
    └─────────────────────────────────────┘
    ↓
    分块参数（可配置）：
    - chunk_size: 1000 字符/块
    - chunk_overlap: 100 字符（相邻块重叠，保持上下文）
    - 分隔符优先：按段落(\n\n)、标题(###)、句子(。)分割

【第4步】生成向量（Embedding）
    ↓
    对每个文本块调用 Embedding API：
    
    chunk_text: "新员工入职需要准备以下材料：身份证、学历证明、离职证明..."
         ↓
    ┌─────────────────────────────────────────┐
    │  Qwen Embedding API                     │
    │  输入：文本块（最多512 tokens）          │
    │  输出：1536维向量 → 截断为768维          │
    └─────────────────────────────────────────┘
         ↓
    vector: [0.023, -0.156, 0.789, ..., 0.445]  ← 768个浮点数
    
    ↓
    重复此过程，直到所有 chunk 都有对应的向量

【第5步】存储到数据库
    ↓
    ┌─────────────────────────────────────────────┐
    │  写入 documents 表（文档元信息）               │
    │  ├─ title: "员工手册"                         │
    │  ├─ filename: "employee_handbook.md"        │
    │  ├─ category: "HR政策"                        │
    │  ├─ embedding_status: "completed"           │
    │  └─ chunk_count: 23                         │
    └─────────────────────────────────────────────┘
    ↓
    ┌─────────────────────────────────────────────┐
    │  写入 document_chunks_with_vectors 表        │
    │  ├─ document_id: 1                          │
    │  ├─ chunk_index: 0                          │
    │  ├─ chunk_text: "新员工入职需要准备..."       │
    │  └─ embedding: [0.023, -0.156, ...]         │
    └─────────────────────────────────────────────┘
    ↓
    ✅ 知识库构建完成！可以开始问答了
```

### 🧮 向量化原理详解

#### 1. 什么是 Embedding？

```
Embedding = 把文本变成「语义坐标」

想象一个 768 维的空间：
- "入职" 在这个空间的某个位置
- "请假" 在另一个位置  
- "离职" 又在另一个位置

语义相近的词，在这个空间中的距离也近：

        维度1   维度2   维度3   ...   维度768
入职    0.5    -0.2     0.8          0.1
报到    0.6    -0.1     0.7          0.2   ← 和"入职"很接近
离职    -0.5    0.3    -0.6         -0.4   ← 和"入职"距离远
```

#### 2. 相似度计算

```
用户问："入职要带什么？"
    ↓ 向量化
query_vector: [0.55, -0.18, 0.75, ..., 0.15]

与数据库中的文档块比较：
┌────────────────────────────────────────────────────┐
│  chunk_0 (公司历史)                                 │
│  vector: [0.1, 0.2, -0.1, ..., 0.3]                 │
│  相似度: 0.23 (23%)  ← 不太相关                      │
├────────────────────────────────────────────────────┤
│  chunk_6 (入职材料)                                 │
│  vector: [0.52, -0.20, 0.77, ..., 0.14]             │
│  相似度: 0.91 (91%)  ← 非常相关！✅                  │
├────────────────────────────────────────────────────┤
│  chunk_7 (第一天安排)                               │
│  vector: [0.48, -0.15, 0.70, ..., 0.18]             │
│  相似度: 0.78 (78%)  ← 相关 ✅                       │
└────────────────────────────────────────────────────┘
```

**计算方式：** 余弦相似度（Cosine Similarity）
```python
similarity = (A · B) / (||A|| × ||B||)
# 结果范围：-1（完全相反）到 1（完全相同）
# 实际使用：0-1 范围，越高越相似
```

#### 3. 为什么用向量而不是关键词？

| 对比项 | 关键词搜索 | 向量搜索 |
|-------|----------|---------|
| "入职" → | 只能找到含"入职"的文档 | 能找到"报到"" onboarding"" 新员工"等 |
| "怎么请假" → | 找不到"如何休假" | 能找到"请假流程""调休申请"等 |
| "工资多少" → | 找不到"薪酬标准" | 能找到"薪资结构""月薪计算"等 |
| 语义理解 | ❌ 无 | ✅ 有 |
| 同义词匹配 | ❌ 无 | ✅ 有 |

```
❌ 简单切分（每1000字符一刀切）：

```

### 🐛 常见问题

**Q1: 上传文档后搜索不到内容？**
```
可能原因：
1. 文档尚未完成向量化（异步处理需要时间）
2. 文档内容太少，分块后语义不明确
3. 查询与文档措辞差异太大

检查方法：
SELECT embedding_status FROM documents WHERE id = ?;
-- 应为 'completed' 才是处理完成
```

**Q2: 向量维度为什么选768？**
```
- 原始模型输出：1536维
- 截断到768维：节省50%存储空间
- 实验验证：768维保留了90%+的语义信息
- 可在 config.py 中调整
```

**Q3: 大文件怎么处理？**
```
建议方案：
- 超大PDF(>100MB)：先拆分成章节再上传
- 图片多的文档：先用OCR提取文字
- 扫描版PDF：启用OCR功能
```

**Q4: 如何更新已有文档？**
```
当前方案：
1. 删除旧文档：DELETE FROM documents WHERE id = ?
2. 重新上传新版本

未来优化：
- 增量更新（只修改变化的部分）
- 版本管理（保留历史版本）
```

## 🗄️ 数据库表结构

### 文档表 (documents)

```sql
CREATE TABLE documents (
    id INT PRIMARY KEY,
    title VARCHAR(200),           -- 文档标题
    filename VARCHAR(255),        -- 原始文件名
    category VARCHAR(50),         -- 分类
    embedding_status VARCHAR(50), -- 向量化状态
    chunk_count INT,              -- 分块数量
    created_at TIMESTAMP
);
```

### 文档分块表 (document_chunks_with_vectors)

```sql
CREATE TABLE document_chunks_with_vectors (
    id INT PRIMARY KEY,
    document_id INT,              -- 关联文档ID
    chunk_index INT,              -- 分块序号
    chunk_text TEXT,              -- 分块文本内容
    embedding VECTOR(768),        -- 向量（768维）
    created_at TIMESTAMP
);
```

## ⚙️ 核心组件

### 1. **向量化工具**（Qwen Embedding）
```
输入：任意文本
处理：把文本转换成 1536 维向量
输出：数学向量
用途：让电脑理解文本的含义
```

### 2. **向量搜索引擎**（PostgreSQL + pgvector）
```
输入：查询向量 + 文档向量库
处理：计算余弦相似度
输出：相关度分数 + 排序
用途：快速找到最相关的文档
```

### 3. **文本生成模型**（豆包 Chat）
```
输入：文档 + 问题 + 指令
处理：AI 理解并生成回答
输出：自然语言文本
用途：生成符合上下文的回答
```

## 🐛 常见问题

### Q1: 为什么有时候检索不到相关文档？
**A:** 可能是：
- 知识库中没有相关内容
- 文档尚未完成向量化
- 查询表达与文档措辞差异大

**解决：**
- 检查文档是否已上传并完成向量化
- 尝试用不同的关键词提问
- 使用更通用的表达方式

### Q2: 置信度低是什么意思？
**A:** 置信度反映回答的可靠程度：
- **> 70%**: 高度相关，回答可靠 ✅
- **30%-70%**: 中等相关，需要核实 ⚠️
- **< 30%**: 低度相关，可能不准确 ❌

**建议：** 置信度低时，建议用户换一种方式提问或补充更多资料。

### Q3: 怎么上传新的知识文档？
**A:** 可以通过 API 上传：
```bash
POST http://localhost:8000/api/documents/upload
Content-Type: multipart/form-data

file: @your_document.pdf
```

支持格式：PDF、Word、TXT、Markdown

### Q4: 能支持多轮对话吗？
**A:** 当前是基础问答，多轮对话增强中：
- 保留对话历史上下文
- 支持追问和澄清
- 关联之前的回答

### Q5: 向量维度为什么是 768？
**A:** 
- Qwen Embedding 原始输出是 1536 维
- 为了节省存储空间，截断到 768 维
- 实验表明 768 维已能保持较好效果

## 💡 为什么要用 RAG？

| 问题 | 传统方式 ❌ | RAG 方式 ✅ |
|------|-----------|----------|
| **精准性** | AI 可能瞎编 | 基于真实文档，减少幻觉 |
| **可溯源** | 无法追踪来源 | 显示明确的信息来源 |
| **及时性** | 需要重新训练 | 直接更新知识库即可 |
| **可控性** | AI 容易出错 | 通过文档内容控制质量 |
| **成本** | 需要大量计算 | 只需要搜索 + 轻量级 AI |

## 🔄 四大能力对比

| 能力 | 核心特点 | 关键技术 | 输出形式 |
|-----|---------|---------|---------|
| **知识问答** (RAG) | 查文档、给答案 | Embedding + 向量检索 + LLM | 回答+来源+置信度 |
| **数据查询** (NL2SQL) | 查数据、给数值 | NL2SQL + SQL执行 | 数据+图表 |
| **智能操作** (A2UI) | 执行指令、改数据 | 参数解析 + 事务执行 | 操作结果+日志 |
| **经营诊断** (BI+AI) | 做分析、给建议 | 指标计算 + 规则检测 + LLM | 诊断报告+建议 |

**RAG 的独特价值：**
- ✅ 减少 AI 幻觉（胡说八道）
- ✅ 可追溯信息来源
- ✅ 知识更新简单（换文档即可）
- ✅ 控制回答质量

## 📝 总结

RAG 系统就是：

1. **理解问题** - 识别知识问答意图
2. **向量化** - 把问题变成向量
3. **找相关资料** - 从知识库快速检索
4. **组织信息** - 构建有上下文的提示词
5. **生成回答** - 让 AI 基于资料回答
6. **显示来源** - 告诉用户信息来自哪里

这样既保证了回答的准确性，又让用户能够追踪信息来源。
