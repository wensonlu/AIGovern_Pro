# 🤖 RAG 系统工作流程指南

> **RAG** = Retrieval Augmented Generation（检索增强生成）

## 🎯 一句话理解 RAG

**用户提问 → 系统自动在知识库找相关资料 → AI 基于资料生成回答 → 告诉用户信息来自哪里**

---

## 📖 通俗类比

想象你在图书馆查资料：

1. **用户提问**："入职信息"
   - 就像你向图书馆员说："我要找关于入职的资料"

2. **系统搜索**（向量检索）
   - 图书馆员不是傻傻地翻书，而是用一种"智能方式"快速定位相关书籍
   - 系统用数学模型（向量）来表示"入职信息"的含义
   - 然后用相似度计算找出最相关的文档

3. **整理资料**
   - 图书馆员找到 5 本相关书籍，按相关程度排序
   - 系统找到 5 个相关文档分块，按相似度排序（45%, 19%, 16%...）

4. **AI 生成回答**
   - 图书馆员把这 5 本书交给一位"智能助手"
   - 助手快速阅读，用自己的话总结答案
   - 系统把文档内容交给 LLM，生成自然流畅的回答

5. **显示来源**
   - 图书馆员告诉你："这是从第 2 和第 4 本书找到的"
   - 系统显示："📚 信息来源：company_policy.md 45% 相关度"

---

## 🔄 完整流程（5 个步骤）

### 第 1️⃣ 步：用户在前端提问

```
【前端对话面板】
用户输入：[入职信息] → 点击发送
          ↓
发送 HTTP 请求到后端
```

**实际网络请求：**
```
POST http://localhost:8000/api/chat
Content-Type: application/json

{
  "question": "入职信息",
  "session_id": "default",
  "top_k": 5
}
```

---

### 第 2️⃣ 步：后端生成"语义向量"

**问题：** "入职信息" 是普通文本，系统怎么理解它的意思？

**解决方案：** 用 AI 把文本转换成数学向量（一组数字）

```
"入职信息"
  ↓
【调用 Qwen 通义千问 Embedding API】
  ↓
[0.123, -0.456, 0.789, ..., 0.012]  ← 1536 个数字表示"入职信息"的含义
  ↓
【截断到 768 维】（因为数据库列是 768 维）
  ↓
[0.123, -0.456, 0.789, ..., 0.012]  ← 768 个数字
```

**类比：** 把"入职信息"这个词转换成一个"指纹"，方便快速比对相似度

---

### 第 3️⃣ 步：在数据库中搜索相似文档

**问题：** 有 11 个文档分块，怎么快速找到最相关的 5 个？

**解决方案：** 用 pgvector（PostgreSQL 的向量插件）计算相似度

```
系统把用户的向量[0.123, -0.456, ...]与数据库中的每个文档向量比对：

文档 1 的向量：[0.120, -0.450, ...] → 相似度 45% ✅ 最相似
文档 2 的向量：[0.100, -0.400, ...] → 相似度 19%
文档 3 的向量：[0.050, -0.200, ...] → 相似度 16%
文档 4 的向量：[-0.5, 0.8, ...]    → 相似度 2%  ❌ 不相关
...
```

**SQL 查询：**
```sql
SELECT dc.id, dc.chunk_text, d.filename,
       1 - (dc.embedding <=> [0.123, -0.456, ...]) as relevance
FROM document_chunks_with_vectors dc
JOIN documents d ON dc.document_id = d.id
ORDER BY relevance DESC
LIMIT 5
```

**结果（返回 5 个最相似的文档）：**
```json
[
  {
    "filename": "company_policy.md",
    "chunk_index": 0,
    "relevance": 0.454,
    "text": "### 新员工入职流程\n1. **收到 Offer 后**..."
  },
  {
    "filename": "company_policy.md",
    "chunk_index": 1,
    "relevance": 0.187,
    "text": "### 五险一金\n- 缴纳比例..."
  },
  ...
]
```

**类比：** 就像找到了 5 本最相关的书，按相关程度排序

---

### 第 4️⃣ 步：AI 基于这些文档生成回答

**问题：** 有了相关文档，系统怎么生成自然的回答？

**解决方案：** 把文档内容交给大模型（豆包），让它总结

```
系统构建一个"提示词"：

---
请基于以下文档回答问题。

文档内容：
[文档 1] ### 新员工入职流程
1. **收到 Offer 后**
   - 签署入职合同
   - 完成背景调查（3-5 个工作日）

[文档 2] 2. **入职当天**
   - 上午 9:00 到达人力资源部
   - 领取员工卡和工作电脑

[文档 3] ...

问题：入职信息

要求：
1. 基于提供的文档内容回答
2. 如果文档中没有相关信息，请说明
---

↓ 发送给豆包 API
↓
【豆包 AI 思考】"用户问入职信息，我有 3 份相关文档..."
↓
【生成回答】"新员工入职需要以下步骤：
1. 收到 Offer 后签署合同
2. 入职当天上午 9 点到达人力资源部
..."
```

**类比：** 就像给一个聪明的助手提供 5 本书，让它用自己的话总结答案

## 📊 关键数据转换

| 格式 | 用途 | 举例 |
|------|------|------|
| **文本** | 原始输入 | "入职信息" |
| **向量** | 语义编码 | [0.123, -0.456, ...×768] |
| **相似度** | 匹配程度 | 0.454 (45.4%) |
| **文档** | 知识内容 | "### 新员工入职流程..." |
| **上下文** | LLM 输入 | "文档 1: 新员工入职流程..." |
| **回答** | AI 输出 | "新员工入职需要..." |
| **JSON** | API 响应 | {answer, sources, confidence} |

## ⚙️ 核心组件

### 1. **向量化工具**（Qwen Embedding）
```
输入：任意文本
处理：把文本转换成 1536 维向量
输出：数学向量
用途：让电脑理解文本的含义
```

### 2. **向量搜索引擎**（PostgreSQL + pgvector）
```
输入：查询向量 + 文档向量库
处理：计算余弦相似度
输出：相关度分数 + 排序
用途：快速找到最相关的文档
```

### 3. **文本生成模型**（豆包 Chat）
```
输入：文档 + 问题 + 指令
处理：AI 理解并生成回答
输出：自然语言文本
用途：生成符合上下文的回答
```

### 用户问："产品保修期是多久？"

```
Step 1: 向量化
"产品保修期是多久？" → [向量]

Step 2: 搜索
在数据库中找最相关的文档...

  - product_specs.md (80% 相似度) ← "产品保修期"相关
  - company_policy.md (10% 相似度) ← 员工保险相关

【文档 1】"产品保修期：24 个月..."

Step 6: 返回结果
{
  "answer": "产品保修期为 24 个月...",

  ]
```

## 💡 为什么要这样做？

| 问题 | 传统方式 ❌ | RAG 方式 ✅ |
|------|-----------|----------|
| **精准性** | AI 可能瞎编 | 基于真实文档，减少幻觉 |
| **可溯源** | 无法追踪来源 | 显示明确的信息来源 |
| **及时性** | 需要重新训练 | 直接更新知识库即可 |
| **可控性** | AI 容易出错 | 通过文档内容控制质量 |
| **成本** | 需要大量计算 | 只需要搜索 + 轻量级 AI |

---

## 📝 总结

RAG 系统就是：

1. **理解问题** - 把问题变成向量
2. **找相关资料** - 从知识库快速检索
3. **组织信息** - 构建有上下文的提示词
4. **生成回答** - 让 AI 基于资料回答
5. **显示来源** - 告诉用户信息来自哪里

这样既保证了回答的准确性，又让用户能够追踪信息来源。
