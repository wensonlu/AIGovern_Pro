# 📈 经营诊断系统工作流程指南

> **BI+AI** = Business Intelligence + Artificial Intelligence（商业智能+人工智能）

## 🎯 一句话理解经营诊断

**用户询问经营状况 → 系统自动计算核心指标 → 基于规则识别异常 → AI深度分析根因 → 生成诊断报告与改进建议**

---

## 📖 通俗类比

想象你有一位专业的企业医生：

1. **用户描述症状**："医生，我最近感觉生意不太好，订单量正常吗？"
   - 就像企业主向经营顾问咨询业务状况

2. **体检数据采集**（指标计算）
   - 医生给你做全面体检：量血压、测心率、验血...
   - 系统查询数据库：订单数、GMV、用户数、转化率...
   - 采集8项核心经营指标

3. **异常指标识别**（规则检测）
   - 医生对比正常值：血压偏高⚠️、心率正常✅...
   - 系统对比阈值：订单数4单⚠️（低于500）、转化率5%✅...
   - 自动标记异常项

4. **深度病因分析**（根因分析）
   - 医生分析：血压高可能是因为饮食不当、缺乏运动...
   - AI分析：订单量低可能是因为营销活动不足、竞争加剧...
   - 调用LLM生成深度分析报告

5. **开处方提建议**（决策建议）
   - 医生建议：低盐饮食、每天运动30分钟、定期复查...
   - 系统建议：开展促销活动、优化产品页面、加强用户运营...
   - 提供3-5条可落地的改进措施

6. **生成体检报告**（诊断报告）
   - 医生给你一份完整的体检报告
   - 系统生成包含指标、问题、根因、建议的完整诊断书

### 第 2️⃣ 步：意图识别（经营诊断 vs 数据查询）

**问题：** "订单量是多少"和"订单量是否正常"有什么区别？

**关键区别：**
| 提问方式 | 意图 | 处理方式 |
|---------|------|---------|
| "订单量是多少？" | data_query | 只返回数字：4单 |
| "订单量正常吗？" | business_diagnosis | 分析判断：⚠️ 偏低（阈值500）|

**解决方案：** 识别诊断类关键词

```
用户输入："订单量是否正常？"
  ↓
【调用 LLM 进行意图识别】
  ↓
分析关键词：
- "订单量" → 指标名称
- "是否" → 判断类问题
- "正常" → 需要对比阈值判断
  ↓
判断结果：business_diagnosis（经营诊断）✅
```

**诊断类关键词库：**
```python
diagnosis_keywords = [
    "是否正常", "怎么样", "情况如何",      # 判断类
    "为什么", "怎么回事", "原因",          # 根因类
    "分析", "诊断", "评估", "体检",        # 分析类
    "建议", "怎么办", "如何改进",          # 建议类
    "下滑", "下降", "不好", "差",          # 问题类
]
```

---

### 第 3️⃣ 步：计算核心业务指标

**问题：** 需要采集哪些经营指标？怎么计算？

**解决方案：** 从数据库实时计算8项核心指标

```python
【指标计算过程】

1. 订单总数 (order_count)
   SQL: SELECT COUNT(*) FROM orders
   → 4单

2. GMV总交易额 (gmv)
   SQL: SELECT SUM(amount) FROM orders
   → 16,972.00元

3. 用户总数 (total_users)
   SQL: SELECT COUNT(*) FROM users
   → 3人

4. 活跃用户 (active_users)
   SQL: SELECT COUNT(DISTINCT user_id) FROM orders
   → 3人

5. 转化率 (conversion_rate)
   计算: active_users / total_users * 100
   → 100.0%

6. 客单价 (avg_order_value)
   计算: gmv / order_count
   → 4,243.00元

7. 商品总数 (product_count)
   SQL: SELECT COUNT(*) FROM products
   → 4个

8. 库存紧张商品 (low_stock_count)
   SQL: SELECT COUNT(*) FROM products WHERE stock < 10
   → 0个
```

**采集的指标清单：**

| 指标名称 | 计算方法 | 诊断价值 |
|---------|---------|---------|
| order_count | COUNT(orders) | 反映业务规模 |
| gmv | SUM(orders.amount) | 反映营收能力 |
| total_users | COUNT(users) | 反映用户基础 |
| active_users | DISTINCT order用户 | 反映用户质量 |
| conversion_rate | 下单用户/总用户 | 反映转化效率 |
| avg_order_value | GMV/订单数 | 反映客单价 |
| product_count | COUNT(products) | 反映商品丰富度 |
| low_stock_count | stock<10的商品 | 反映供应链健康度 |

---

### 第 4️⃣ 步：异常检测（规则引擎）

**问题：** 怎么判断指标是否正常？

**解决方案：** 基于业务规则的健康度检查

```python
【异常检测规则】

1. 订单数检查
   规则: order_count < 500
   当前值: 4
   结果: 🔴 异常（低于阈值496单）
   严重级别: high
   
2. 转化率检查
   规则: conversion_rate < 2.0%
   当前值: 100.0%
   结果: 🟢 正常
   严重级别: -

3. 活跃用户检查
   规则: active_users < 3000
   当前值: 3
   结果: 🔴 异常（低于阈值2997人）
   严重级别: high
```

**检测结果：**
```json
{
  "total_issues": 2,
  "issues": [
    {
      "issue": "订单数下降",
      "severity": "high",
      "current": 4,
      "threshold": 500
    },
    {
      "issue": "活跃用户数下降",
      "severity": "high",
      "current": 3,
      "threshold": 3000
    }
  ]
}
```

**阈值设定依据：**

| 指标 | 阈值 | 设定逻辑 |
|-----|------|---------|
| order_count | 500 | 日均订单基准线 |
| conversion_rate | 2.0% | 行业平均水平 |
| active_users | 3000 | 活跃用户基准线 |

---

### 第 5️⃣ 步：AI 深度分析（根因+建议）

**问题：** 发现异常后，如何分析原因和给出建议？

**解决方案：** 调用大模型进行智能分析

#### 5.1 根因分析

```python
【调用 LLM 进行根因分析】

Prompt:
作为一个数据分析专家，分析以下经营问题的根本原因。

问题：订单数下降

背景数据：
- 订单总数: 4单
- GMV: 16972.00元
- 活跃用户: 3人
- 转化率: 100.0%
- 客单价: 4243.00元

请提供：
1. 问题的根本原因
2. 影响范围
3. 应采取的措施

LLM 返回:
"订单量偏低的根本原因分析：

1. 根本原因：
   - 用户基数过小（仅3人），即使100%转化也无法产生大量订单
   - 缺乏有效的用户获取渠道和营销活动
   - 可能处于业务初期阶段，品牌知名度不足

2. 影响范围：
   - 直接影响GMV增长天花板
   - 难以形成规模效应，单位成本较高
   - 影响供应链议价能力和库存周转

3. 应采取的措施：
   - 加大用户获取投入，扩大用户基数
   - 开展新用户优惠活动，降低首次购买门槛
   - 建立用户推荐机制，利用现有用户拉新"
```

#### 5.2 决策建议生成

```python
【调用 LLM 生成改进建议】

Prompt:
基于以下分析，为企业提供可行的改进建议。

分析结果：
{根因分析内容}

要求：
1. 提供 3-5 个具体的改进建议
2. 每个建议说明优先级和预期效果
3. 建议要可落地执行

LLM 返回:
"改进建议：

🔴 高优先级：
1. 开展新用户专享活动
   - 首单立减50元，降低试用门槛
   - 预期：新用户增长30-50%

2. 建立用户推荐奖励机制
   - 老用户推荐新用户成功下单，双方各得20元券
   - 预期：月均新增用户100+

🟡 中优先级：
3. 优化产品页面体验
   - 简化购买流程，减少跳转步骤
   - 预期：转化率提升5-10%

4. 加强社交媒体运营
   - 在微信公众号、小红书发布产品内容
   - 预期：提升品牌曝光度"
```

## 🎯 支持的诊断类型

### 1. 单项指标诊断
```
用户："订单量正常吗？"
系统：分析订单数，对比阈值，返回判断结果和建议
```

### 2. 用户活跃度诊断
```
用户："用户活跃度怎么样？"
系统：分析总用户、活跃用户、转化率，评估用户健康度
```

### 3. 营收能力诊断
```
用户："GMV表现如何？"
系统：分析GMV、客单价、订单量，评估营收状况
```

### 4. 转化率诊断
```
用户："转化率是否正常？"
系统：计算转化率，对比行业平均，分析转化漏斗
```

### 5. 综合经营体检
```
用户："整体经营状况如何？"
系统：全面分析所有指标，生成完整诊断报告
```

## 🐛 常见问题

### Q1: 诊断结果为什么有时不准确？
**A:** 可能是：
- 数据量太小，指标波动大（如只有几个订单）
- 阈值设定与业务阶段不匹配
- 缺少历史数据对比

**解决：** 
- 积累更多数据后再诊断
- 根据业务阶段调整阈值
- 结合人工判断

### Q2: 能自定义诊断阈值吗？
**A:** 当前阈值是硬编码的，后续会支持：
- 通过配置文件调整阈值
- 基于历史数据自动学习阈值
- 按行业/业务阶段设定不同标准

### Q3: 怎么查看历史诊断记录？
**A:** 目前每次诊断是独立的，暂不支持历史对比。可以通过 API 查看：
```http
GET http://localhost:8000/api/diagnosis/summary
GET http://localhost:8000/api/diagnosis/metrics
```

### Q4: 诊断能预测未来趋势吗？
**A:** 当前是描述性分析，预测功能在规划中：
- 基于历史数据的趋势预测
- 异常预警（提前发现问题）
- 智能预警推送

## 🔄 四大能力对比

| 能力 | 核心特点 | 数据流向 | 智能程度 |
|-----|---------|---------|---------|
| **知识问答** (RAG) | 查文档、给答案 | 文档 → 向量检索 → 生成 | ⭐⭐⭐ |
| **数据查询** (NL2SQL) | 查数据、给数值 | SQL → 数据库 → 展示 | ⭐⭐ |
| **智能操作** (A2UI) | 执行指令、改数据 | 解析 → 执行 → 日志 | ⭐⭐⭐ |
| **经营诊断** (BI+AI) | 做分析、给建议 | 指标 → 分析 → 建议 | ⭐⭐⭐⭐ |

**经营诊断的独特价值：**
- ✅ 不只是给数据，而是给**判断**（是否正常）
- ✅ 不只是查现状，而是给**原因**（为什么）
- ✅ 不只是看问题，而是给**方案**（怎么办）
