# ReAct Prompt 设计

本文档详细设计 ReAct 修正引擎的 Prompt 策略，针对不同验收失败类型制定对应的修复引导。

## 一、失败类型分类

```
┌─────────────────────────────────────────────────────────────┐
│                      失败类型分类树                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  BUILD_ERROR                                               │
│  ├── 编译失败 (CompilationError)                           │
│  ├── 依赖冲突 (DependencyConflict)                         │
│  ├── 构建失败 (BuildFailed)                                │
│  └── 服务启动失败 (ServiceStartFailed)                     │
│                                                             │
│  STATIC_ERROR                                              │
│  ├── 布局偏差 (LayoutMismatch)                              │
│  ├── 样式差异 (StyleMismatch)                               │
│  ├── 元素缺失 (ElementMissing)                              │
│  └── 响应式问题 (ResponsiveIssue)                           │
│                                                             │
│  DYNAMIC_ERROR                                            │
│  ├── 交互失败 (InteractionFailed)                          │
│  ├── 流程中断 (FlowBroken)                                  │
│  ├── 数据错误 (DataError)                                  │
│  └── 权限问题 (PermissionDenied)                            │
│                                                             │
│  SEMANTIC_ERROR                                           │
│  ├── 业务逻辑错误 (BusinessLogicError)                      │
│  ├── 数据不一致 (DataInconsistency)                         │
│  ├── 规则校验失败 (RuleValidationFailed)                    │
│  └── 边界条件错误 (BoundaryConditionError)                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、基础 Prompt 模板

### 2.1 统一系统提示

```
你是一个智能代码修复助手，专门解决 AI 生成代码的验收失败问题。

## 核心原则
1. 每次修复只针对一个问题，不要试图一次修复多个问题
2. 修复后必须验证，确保问题真正解决
3. 如果一次修复无效，分析原因后尝试其他策略
4. 记录每次修复的尝试和结果，形成学习数据

## 修复上限
- 同一问题最多修复 3 次
- 超过次数则标记为"需要人工介入"

## 输出格式
请按以下格式输出修复结果：
```
## 失败分析
[简述失败原因]

## 修复策略
[描述具体修复方案]

## 修改内容
```diff
[展示代码改动]
```

## 验证计划
[描述如何验证修复是否成功]
```
```

### 2.2 失败类型-specific Prompt

#### BUILD_ERROR - 编译失败

```
## 场景
代码编译失败，错误信息如下：
```
{error_message}
```

## 分析要点
1. 仔细阅读编译器错误信息，定位具体文件和行号
2. 判断是语法错误、类型错误还是导入错误
3. 检查 AI 生成的代码是否符合目标语言规范
4. 查看是否有遗漏的依赖或类型定义

## 修复策略
- 语法错误：修正语法结构
- 类型错误：补充类型定义或类型转换
- 导入错误：添加缺失的 import 或检查路径
- 依赖问题：更新依赖配置

请给出具体的修复方案。
```

#### BUILD_ERROR - 依赖冲突

```
## 场景
依赖版本冲突，错误信息如下：
```
{error_message}
```

## 分析要点
1. 识别冲突的依赖包和版本
2. 分析项目其他依赖的版本兼容性
3. 查找合适的兼容版本组合
4. 考虑是否需要调整其他依赖版本

## 修复策略
- 升级/降级冲突依赖到兼容版本
- 使用版本范围而非固定版本
- 排除传递依赖中的冲突版本

请给出具体的修复方案。
```

#### STATIC_ERROR - UI 布局偏差

```
## 场景
UI 静态验收失败，截图与设计稿对比发现布局偏差。

## 失败详情
- 期望位置: {expected_position}
- 实际位置: {actual_position}
- 偏差值: {deviation}
- 设计稿截图: {design_screenshot}
- 实际截图: {actual_screenshot}

## 分析要点
1. 分析是绝对定位问题还是相对布局问题
2. 检查 CSS 布局相关属性（position, flex, grid）
3. 查看是否父容器影响了子元素布局
4. 对比设计稿的尺寸和间距

## 修复策略
- 调整 CSS 定位属性
- 修改 flex/grid 布局参数
- 修正尺寸和间距数值
- 检查响应式断点设置

请给出具体的修复方案。
```

#### STATIC_ERROR - 样式差异

```
## 场景
UI 静态验收失败，发现样式与设计稿不符。

## 失败详情
- 差异类型: {style_type} (颜色/字体/间距/圆角/阴影等)
- 期望值: {expected_value}
- 实际值: {actual_value}
- 元素: {element_selector}

## 分析要点
1. 对比设计稿和实现的具体样式值
2. 检查是否有样式被覆盖
3. 查看是否使用了错误的 CSS 变量
4. 确认是否缺少必要的样式属性

## 修复策略
- 修正具体的样式属性值
- 检查优先级并正确设置样式
- 使用正确的 CSS 变量或主题
- 补充缺失的样式属性

请给出具体的修复方案。
```

## 三、修复验证 Prompt

### 3.1 验证基础模板

```
## 验证任务
请验证修复是否成功。

## 原始错误
```
{original_error}
```

## 修复内容
```diff
{fix_diff}
```

## 验证要求
1. 重新执行验收测试
2. 检查原错误是否已消除
3. 确保没有引入新问题
4. 记录验证结果

## 输出格式
```
验证结果: PASS / FAIL
失败原因: [如果 FAIL，说明原因]
下一步建议: [如果 FAIL，建议如何继续修复]
```
```

### 3.2 分层验证提示

```
## 构建验证
重新执行构建，确认：
- 编译通过无错误
- 依赖解析成功
- 服务可以启动

## 静态验证
重新执行静态验收，确认：
- UI 元素位置正确
- 样式与设计稿一致
- 响应式布局正常

## 动态验证
重新执行动态验收，确认：
- 所有交互操作正常
- 业务流程完整
- 数据正确展示

## 语义验证
重新执行语义验收，确认：
- 业务逻辑正确
- 数据一致
- 规则校验通过
```

## 五、特殊场景处理

### 5.1 修复失败升级

```
## 修复状态
你已经尝试了 {attempt_count} 次修复，但仍未能解决问题。

## 升级策略
请执行以下操作：
1. 总结所有尝试过的修复方案
2. 分析失败的根本原因
3. 判断是否需要人工介入
4. 如果需要人工介入，清晰描述问题等待人工处理

## 升级条件
- 尝试次数 >= 3
- 修复方向不明确
- 涉及复杂的架构问题
- 需要业务方确认需求
```

### 5.2 复合问题处理

```
## 问题分析
当前失败可能是由多个问题复合导致：

问题1: {issue_1}
问题2: {issue_2}
问题3: {issue_3}

## 处理策略
1. 先修复最基础的问题（如构建错误）
2. 逐层向上修复
3. 每修复一个维度后重新验证
4. 避免一次性修改太多导致难以定位问题

请制定分步修复计划。
```

## 六、Prompt 优化迭代

### 6.1 反馈收集

每次修复后记录：
- 成功/失败状态
- 修复耗时
- 使用的策略
- 失败原因分析

### 6.2 策略进化

基于反馈数据：
- 识别高成功率的修复策略
- 标记需要避免的修复方向
- 更新 prompt 模板
- 补充新的修复模式
