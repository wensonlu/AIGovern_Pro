# B端管理系统 AI 自主验收技术方案

## 一、系统特性分析

| 特性 | 对验收体系的影响 |
|------|------------------|
| 表单/表格为主 | 动态验收重点在 CRUD 操作、批量操作、数据校验 |
| 审批流复杂 | 语义验收需覆盖流程状态、权限校验、节点跳转 |
| 权限体系严格 | 需校验 RBAC、数据权限、行级权限 |
| 稳定优先 | 回归测试覆盖度要求高 |
| 企业级集成 | 涉及第三方 API、消息队列、数据同步 |

## 二、验收架构设计

### 2.1 整体架构图

```
┌─────────────────────────────────────────────────────────────────┐
│                      任务调度层 (Temporal)                       │
├─────────────────────────────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐            │
│  │ 构建验收 │→│ 静态验收 │→│ 动态验收 │→│ 语义验收 │            │
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘            │
│       │            │            │            │                 │
│       └────────────┴─────┬──────┴────────────┘                 │
│                          ↓                                      │
│                   ┌─────────────┐                               │
│                   │ ReAct 修正环 │                               │
│                   └─────────────┘                               │
├─────────────────────────────────────────────────────────────────┤
│                      执行层 (Playwright + LLM)                   │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 验收流程

```
需求输入 → 用例自动生成 → 构建打包 → UI 静态校验 → 
交互动态校验 → 业务语义校验 → 失败? → ReAct修正 → 
重新验收 → 通过 → 交付
```

## 三、验收标准设计

### 3.1 构建验收（前端+后端）

| 检查点 | 技术实现 |
|--------|----------|
| 前端构建成功 | webpack/vite 构建无错误 |
| 后端编译通过 | mvn gradle 编译通过 |
| 依赖无冲突 | 依赖树分析，无版本冲突 |
| 镜像构建成功 | Dockerfile 构建 + 推送 |
| 服务启动成功 | 健康检查接口返回 200 |

### 3.2 静态验收（B端 UI 特点）

| 检查点 | 技术实现 |
|--------|----------|
| 表格渲染正确 | 列配置、数据分页、排序图标 |
| 表单布局规范 | 标签对齐、输入框高度、必填标识 |
| 按钮状态正常 | 新增/编辑/删除/提交 按钮可点击 |
| 弹框样式一致 | 标题、关闭按钮、遮罩层 |
| 响应式适配 | 1024px / 1280px / 1920px 多尺寸 |
| 企业风格一致 | 颜色、品牌元素、字体层级 |

### 3.3 动态验收（B端核心场景）

#### 3.3.1 CRUD 操作的完整流程

| 场景 | 验收要点 |
|------|----------|
| 新增保存 | 表单校验 → 提交 → 列表新增 → 数据正确 |
| 编辑保存 | 预填数据 → 修改 → 保存 → 列表更新 |
| 删除确认 | 二次确认弹框 → 删除 → 列表移除 |
| 批量操作 | 勾选多条 → 批量删除/启用 → 状态更新 |
| 分页切换 | 翻页 → 数据加载 → 页码同步 |
| 搜索查询 | 关键字 → 请求 → 结果高亮 → 分页正确 |

#### 3.3.2 权限相关场景

| 场景 | 验收要点 |
|------|----------|
| 按钮级权限 | 无权限用户看不到/点不了按钮 |
| 数据级权限 | 只能看到有权限的数据行 |
| 菜单级权限 | 无权限菜单不显示 |
| API 级权限 | 越权请求返回 403 |

#### 3.3.3 审批流场景

| 场景 | 验收要点 |
|------|----------|
| 提交审批 | 状态流转、待办任务生成 |
| 审批通过 | 状态更新、通知发送 |
| 审批驳回 | 状态回退、驳回理由记录 |
| 流程跳转 | 节点流转、历史记录完整 |

### 3.4 语义验收（B端业务逻辑）

| 检查点 | 技术实现 |
|--------|----------|
| 数据一致性 | 数据库状态与前端展示一致 |
| 跨模块校验 | 订单→库存→财务 数据贯通 |
| 业务规则校验 | 促销计算、库存扣减、余额变动 |
| 事务完整性 | 多表操作原子性、分布式事务 |
| 异常边界处理 | 空数据、网络错误、超时处理 |

#### 4.2.3 ReAct 修正引擎

```java
class ReActCorrectionEngine {
    // 失败分类
    // - BUILD_ERROR: 编译/构建问题
    // - UI_MISMATCH: 静态验收失败
    // - INTERACTION_FAIL: 动态验收失败
    // - BUSINESS_ERROR: 语义验收失败
    
    // 修复策略映射
    // BUILD_ERROR → 修复依赖/版本/配置
    // UI_MISMATCH → 调整样式/CSS/组件
    // INTERACTION_FAIL → 重写交互逻辑
    // BUSINESS_ERROR → 修正业务规则
}
```

## 五、实施路线

### 第一阶段：基础能力（1-2个月）

- [ ] CI/CD 流水线搭建（构建验收）
- [ ] Playwright 基础环境搭建
- [ ] 核心 CRUD 场景动态验收

### 第二阶段：验收完善（2-3个月）

- [ ] 静态验收接入（Applitools）
- [ ] 权限场景覆盖
- [ ] 审批流验收
- [ ] ReAct 修正循环

### 第三阶段：智能进化（3-6个月）

- [ ] 用例自动生成
- [ ] 语义验收深化
- [ ] 失败案例学习

## 七、风险与对策

| 风险 | 对策 |
|------|------|
| LLM 生成用例质量不稳定 | 人工抽检 + 反馈优化 |
| 动态验收 flaky | 重试机制 + 环境隔离 |
| 执行时间过长 | 增量验收 + 并行化 |
| 维护成本高 | 用例版本管理 + 复用 |
