| 常见场景         | 传统耗时     | AI方案     | 节省         | 机会成本       |
| ------------ | -------- | -------- | ---------- | ---------- |
| 需求评审反复改      | 3-5天     | 1天       | 3-4天       | 延迟上线       |
| 等设计稿出来       | 1-3天     | 30分钟     | 20-50小时    | 快速验证想法     |
| 原型反复调整       | 1周       | 2小时      | 5天         | 尽早发现可行性问题  |

**现实：你花在"沟通协调+反复修改"上的时间，往往比"实际做事"还多**

### 这套方法能帮你：

- ✅ **缩短2-3倍** — 需求到MVP的周期从4-6周变成5-7天
- ✅ **减少返工** — 早期验证想法是否可行，不走冤枉路
- ✅ **团队聚焦** — 减少内部扯皮，人人有一份清晰的需求
- ✅ **抢占市场** — 快速迭代，在对手还在评审的时候已经上线

## 核心流水线：从模糊想法到可验证MVP

```
① 需求分析         ② 可视化设计         ③ MVP代码
   头脑风暴          流程图+架构           自主开发
   Prompt: 5why     Prompt: mermaid      Prompt: 详需求
   产出: PRD         产出: 设计稿         产出: 可运行版本
   ↓                ↓                    ↓
   1天             2小时               1周
```

### 不同场景的快速选择


| 场景     | 用哪一步 | 投入时间 | 核心问题        |
| ------ | ---- | ---- | ----------- |
| 不确定方向  | 仅做①② | 1.5天 | 这个想法到底靠不靠谱？ |
| 需要快速验证 | ①②③  | 5-7天 | 用户的反应怎么样？   |
| 要正式立项  | 完整流程 | 1-2周 | 做成生产级别的产品   |

## Step 1：需求分析 — 从模糊想法到清晰需求（1天）

**核心方法**：用AI做"五问法"，逐层挖掘问题根因

头脑风暴： SKII Brainstorming

### 实战流程

**第1步：抛出问题**

找一个ChatGPT/Claude，输入这个Prompt：

```
我是[公司]的[岗位]
遇到了[问题描述]
用户是[具体画像]
目标是[提升哪个指标]

请帮我分析：
1. 这个问题的根本原因是什么？（用五问法）
2. 用户有哪些未被满足的需求？
3. 有3个解决方案，哪个优先级最高？为什么？
4. 要验证这个方案有效，应该怎么衡量？
```

**第2步：得到PRD初稿**

AI会输出：

- ✅ 问题拆解
- ✅ 需求优先级
- ✅ 用户故事
- ✅ 验收标准（关键！）

**第3步：优化内容**

让AI继续：

```
根据上面的分析，帮我组织成一份能直接给开发看的PRD
包含：
- 功能描述
- 用户路径（正常+异常）
- 验收标准
- 非功能需求（性能/安全/兼容性）
```

**实际案例**

企业知识库检索问题（基于 AIGovern Pro）：

| 问题        | AI分析              | 优先级     |
| --------- | ------------------- | ------- |
| 知识分散难查找  | 员工查询响应慢 15-30 分钟 | 1️⃣ 先做 |
| 常见问题重复问  | 支持成本高，每天 100+ 重复问   | 2️⃣ 再做 |
| 知识库冷启动   | 新人培训周期长         | 3️⃣ 后做 |


### ⚠️ 常见错误


| ❌ 错误做法         | ✅ 正确做法      |
| -------------- | ----------- |
| "帮我写个需求" → 太笼统 | 明确背景+数据+目标  |
| Prompt越长越好     | 3个段落+4个问题足够 |
| 一次OK就上线        | AI输出要经过开发评审 |

## Step 2：可视化设计 — 从需求到流程图（2小时）

**核心价值**：让团队看到同一个"系统长什么样"

### 实战流程

**第1步：选工具**


| 工具           | 特点          | 推荐     |
| ------------ | ----------- | ------ |
| ProcessOn AI | 中文、国产、流程图专业 | ⭐⭐⭐ 首选 |
| 用Claude描述    | 生成Mermaid代码 | ⭐⭐ 备选  |


**第2步：生成流程图**

打开ProcessOn，输入：

```
帮我画[功能名]的业务流程图

包含这些角色的路径：
- 普通用户（主流程）
- 管理员（特殊操作）
- 系统（后台处理）

异常情况：
- 验证失败
- 权限不足
- 超时处理

用泳道图标示不同角色的操作
```

**第3步：导出给开发**

- ProcessOn导出为PNG/PDF
- 发给开发评审一遍（10分钟）
- 开发提出"技术上做不了"的点 → 调整

### 💡 判断标准

**可以跳过这一步的情况**：

- 功能是单个页面的修改（直接做原型）
- 流程简单到不需要画（几个if-else）

**必须做的情况**：

- 多个系统的交互
- 用户流程超过5个步骤
- 团队对需求理解不一致

## Step 3：原型设计 — 从需求到设计稿（30分钟）

**核心价值**：看到实际的UI界面，比用PPT讲更清楚

### 实战流程

**第1步：用Figma或Claude画原型**

如果你有现成的设计工具（Figma），直接用AI生成Prompt让设计师参考。

如果没有，用Claude生成HTML原型：

```
基于这个需求：[需求描述]

用HTML+CSS生成一个可交互的原型

要求：
- 响应式设计（手机+电脑都能看）
- 体现核心用户流程
- 用真实数据，不用占位符
- 可以点击跳转到下一步

颜色风格：[简洁/企业/亮色/暗黑]
```

**第2步：快速评审（10分钟）**

- 在Figma/HTML预览中快速看一眼
- 用户反馈三要素：布局对不对 / 流程清不清楚 / 还缺什么
- 90%的问题在这一步就能发现

**第3步：修改+冻结**

- 根据反馈修改（最多改3轮）
- 定版后就给开发，不再改设计

### ⚠️ 常见错误


| ❌ 错误             | ✅ 正确             |
| ---------------- | ---------------- |
| 原型改10轮，还在打磨细节    | 原型的价值是验证思路，不是完美  |
| 等设计完美后才给开发看      | 开发尽早参与，提出技术限制    |
| 原型做得太炫，开发觉得你要求太高 | 原型只是"参考"，最终由开发判断 |

## Step 4：MVP开发 — 从设计到代码（1周）

**核心价值**：验证用户是否真的需要这个功能，不是做"完整产品"

### 实战流程

**第1步：定义MVP范围**

最常见的错：功能一大堆，以为这样更有说服力。

正确做法：

```
核心假设：用户会[做某事]
MVP目标：只验证这一点
最小化范围：
  ✅ 做：用户主流程
  ❌ 不做：边界情况、性能优化、完整UI
  ❌ 不做：后台管理、数据分析、推送通知
```

**例子**


| 假设        | MVP就是         | 不要加的           |
| --------- | ------------- | -------------- |
| 用户愿意为工具付费 | 产品展示+支付按钮     | 完整支付、发票、客服     |
| 用户喜欢协作功能  | 能邀请朋友+共享一个doc | 权限系统、版本控制、审计日志 |
| 运营想降低审批时间 | 能提交+批准+结果反馈   | 复杂的流程定制、数据报表   |


**第2步：给开发输入需求**

用Claude生成一份"简洁但完整"的需求文档：

```
角色：你是一个技术方案设计师
任务：根据MVP范围，写一份1周内能做完的技术方案

需求：[你的MVP描述]

输出包括：
- 前端界面清单（几个页面？）
- 后端API清单（几个接口？）
- 数据库设计（几张表？）
- 第三方依赖（需要什么？）
- 技术栈建议
- 关键技术风险

约束：
- 假设有1个全栈工程师
- 时间窗口1周
- 用最熟悉的技术栈
```

开发看完应该能说出："这个我1周能做"或"这个得改范围"

**第3步：开发做，你监督**

不需要你天天盯着，但要：

- Day 1-2：后端API搭好
- Day 3-4：前端对接
- Day 5-6：集成测试
- Day 7：上线

中间发现做不完？立即砍功能，不是加班

### ⚠️ MVP失败的3个信号


| 信号         | 解决            |
| ---------- | ------------- |
| "这个得再加..." | 停，就这样上线       |
| "用户说不够"    | 等他们给数据再加，不是感觉 |
| "开发说需要2周"  | 范围太大，砍50%     |

## 快速启动：选择你的路径

根据你的时间和目标，选一条路：

### 路径A：我只有1天（验证想法是否靠谱）

```
Step 1: 需求分析（4小时）✅
  └─ 输出：PRD

Step 2: 可视化设计（2小时）✅
  └─ 输出：流程图/原型

完成：你清楚这个方向对不对了
```

**何时用**：

- 新想法，还不确定
- 给老板/投资人快速验证
- 评估技术可行性

### 路径B：我有1周（快速上线）

```
Step 1: 需求分析（1天）✅
Step 2: 可视化设计（2小时）✅
Step 3: 开发MVP（5天）✅
  └─ 输出：可运行的版本

完成：MVP上线，收集真实用户反馈
```

**何时用**：

- 功能简单清晰
- 想抢先发布
- 有明确的小范围目标

## 实战案例：构建"企业知识库AI助手"

### 原始需求

> "我们员工每天花太多时间查文档。能不能做一个AI助手，员工问问题就能快速找到答案"

### 按流程走下来

**Step 1（1天）**：需求分析

#### Prompt 1：抛出问题

```
我是某制造企业的知识管理负责人

遇到的问题：
- 销售、运营、技术人员每天浪费大量时间翻阅 Wiki/文档/培训资料寻找答案
- 常见问题重复被问100+ 次/天（产品规格、流程审批、常见故障排查）
- 知识分散在 10+ 个系统（JIRA、Wiki、共享盘、邮件存档），搜索成本高

用户画像：销售（需要快速查产品规格）、运营（需要查流程规范）、技术支持（需要快速定位故障）

目标指标：将知识查询平均响应时间从 15-30 分钟降至 2-3 分钟

请帮我分析：
1. 这个问题的根本原因是什么？（用五问法）
2. 用户有哪些未被满足的需求？
3. 有 3 个解决方案，哪个优先级最高？为什么？
4. 要验证这个方案有效，应该怎么衡量？
```

#### Prompt 2：得到PRD初稿

```
根据上面的分析，帮我组织成一份能直接给开发看的 PRD

包含：
- 功能描述（要解决什么问题）
- 用户路径（销售/运营/技术支持各自的使用流程）
- 异常流程处理（知识库空白、答案不准确）
- 验收标准（如何衡量成功）
- 非功能需求（性能/安全/兼容性）
```

#### Prompt 3：优化为技术方案

```
基于上述 PRD，帮我生成一份"技术方案设计"

输出包括：
- 前端界面清单（AGUI 对话面板 + 何种操作引擎）
- 后端 API 清单（几个接口？）
- 数据库设计（RAG 向量库选型、知识入库流程）
- 第三方依赖（向量库、大模型 API）
- 关键技术风险（幻觉问题、答案准确性）

约束：
- 假设 1 个全栈工程师 + 1 个 AI 工程师
- 时间窗口 2 周
- 用 FastAPI + Milvus + DashScope 大模型
```

**Step 2（2小时）**：流程图与原型

```
用户提问 → AI 意图识别 → RAG 检索相关知识 → 大模型生成答案 → 返回结果 + 溯源文档
                                          ↓异常：无相关知识 → 显示"无法回答"，引导人工反馈
```

**Step 3（2周）**：MVP开发

```
Week 1
  Day 1-2: 后端框架 + RAG 流水线搭建
    - FastAPI 应用框架
    - 大模型 + 向量库集成
  
  Day 3-4: 知识入库 + 前端对接
    - 文档分词入库到 Milvus
    - AGUI 对话面板集成
  
  Day 5: 集成测试
    - 端到端流程验证
    - 3 个部门试用

Week 2
  Day 1-2: 答案质量优化
    - 提示词微调
    - 幻觉问题修复
  
  Day 3-4: 权限管控 + 审计
    - 知识库访问控制
    - 查询日志记录
  
  Day 5: 上线

成功标准：
- 查询响应时间 < 3 秒
- 答案准确度 > 85%
- 3 个部门反馈满意度 > 4/5
```

### 结果

- Week 2 周五上线
- Week 3 周一各部门开始使用
- Month 1 数据：查询时间从 20 分钟 → 2 分钟 | 支持成本下降 60%

## 这套方法不适合的场景


| 场景               | 原因            | 建议            |
| ---------------- | ------------- | ------------- |
| 大型系统架构（拆成10+个模块） | 需要技术方案评审      | 用这套方法做小模块，再整合 |
| 高并发系统（日活100万+）   | 需要性能设计        | 先MVP验证，后续再优化  |
| 需要复杂数据分析         | 不是核心功能，容易过度设计 | MVP不做，用户反馈后再加 |
| 涉及政策/法规（如支付/医疗）  | 不能快速迭代        | 先搞清规则，再用这套方法  |

## 最后3条建议

**1. 从最小的功能开始**

不要想"一口吃成大胖子"。从"一个用户用一个功能"开始。

✅ "销售能快速生成报价" → 1周MVP
❌ "完整的CRM系统" → 3个月都做不完

**2. 每一步都找开发反馈**

不要等所有步骤都完成后才给开发。最好的时间是Step 2结束。

开发说"技术做不了"？现在改还来得及。

**3. 记录"为什么这样"**

3个月后你会忘记为什么要做这个功能。

把Step 1输出的"核心问题"贴在Slack置顶，给全team看。
